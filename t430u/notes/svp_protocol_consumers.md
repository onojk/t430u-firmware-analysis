# SVP Subsystem Cross-Reference

Generated by binary search of the extracted BIOS dump for protocol GUIDs in DXE dependency sections.

## Stage 1 password hash protocol — `E3ABB023-B8B1-4696-98E1-8EEDC3D3C63D`

Consumed by 3 modules:
- `LenovoHpmDxe.efi` — Hard Disk Password Manager
- `LenovoSetupSecurityDxe.efi` — Security setup screen (BIOS setup UI)
- `LenovoPasswordCp.efi` — Password character processing (analyzed)

These are all credential-entry contexts. The hash protocol takes a typed password
and produces an intermediate digest.

## Stage 2 password hash protocol — `E01FC710-BA41-493B-A919-53583368F6D9`

Not declared as a load-time dependency by any module.

This is consistent with the protocol being SMM-resident: SMM protocols are
discovered via LocateProtocol at runtime through the SMM communication mechanism,
not declared in DXE depexes. The actual implementation likely lives in
`LenovoSvpManagerSmm.efi` or a similarly-named SMM driver.

## What this reveals architecturally

The SVP isn't a UI feature — it's the root of trust for nearly every privileged
operation on this machine. TPM operations, theft-tracking persistence, signed
firmware updates, scripted enterprise configuration, and manufacturing-mode
operations all consult the SVP service to authorize their action.

Re-flashing the BIOS via Lenovo's tools (without external SPI access) requires
SVP authorization — this list shows why: `LenovoSecureUpdateDxe` is on it.
